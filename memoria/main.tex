% ── 1. Motivación ─────────────────────────────────────────────────────
\section{Motivación del Proyecto}

Los streamers actuales controlan sus efectos visuales mediante teclado, botones físicos o dispositivos como StreamDeck~\cite{elgato_streamdeck} durante sus transmisiones en directo. Esta dependencia en dispositivos físicos causa que deban redirigir su atención causando incomodidates como:

\begin{itemize}
    \item \textbf{Ruptura del contacto visual}: el streamer debe apartar la vista de la cámara para localizar y pulsar el botón correcto en momentos clave.
    \item \textbf{Pérdida de autenticidad}: la reacción genuina queda interrumpida mientras se busca el control, restando naturalidad al contenido.
    \item \textbf{Complejidad técnica}: las producciones más elaboradas requieren un técnico adicional para gestionar efectos en tiempo real.
    \item \textbf{Barrera de acceso}: equipo especializado como el StreamDeck supone una inversión económica que no está al alcance de todos los creadores de contenido.
\end{itemize}

La disponibilidad de APIs de visión computacional de alto rendimiento en el navegador como Google MediaPipe \cite{mediapipe} y de reconocimiento de voz nativo como Web Speech API \cite{webspeech} abre la posibilidad de crear interfaces gestuales multimodal sin hardware adicional. Es este problema el que motiva el desarrollo de MoveStream.

% ── 2. Objetivo ───────────────────────────────────────────────────────
\section{Objetivo del Proyecto}

Desarrollar un prototipo de asistente de efectos visuales para streaming que permita activar overlays y efectos en tiempo real mediante gestos de mano, expresiones faciales y comandos de voz, sin tocar ningún dispositivo durante la transmisión.

Los objetivos específicos son:
\begin{itemize}
    \item Implementar reconocimiento de gestos de mano en tiempo real con MediaPipe HandLandmarker.
    \item Detectar expresiones faciales mediante MediaPipe FaceLandmarker y análisis de blendshapes.
    \item Integrar reconocimiento de voz con la Web Speech API, condicionado a un gesto activador.
    \item Renderizar efectos visuales y overlays sobre el vídeo de la cámara usando Three.js \cite{threejs}.
\end{itemize}

La aplicación se ejecuta íntegramente en el navegador web, sin backend, usando únicamente una webcam estándar y el micrófono del sistema.

% ── 5. Descripción ────────────────────────────────────────────────────
\section{Descripción de la Propuesta Desarrollada}

\subsection{Stack Tecnológico}

MoveStream es una aplicación web sin backend construida con React 18~\cite{react} y Vite~\cite{react}. El stack completo es:

\begin{itemize}
    \item \textbf{React 18 + Vite}: scaffolding, UI declarativa y gestión de hooks.
    \item \textbf{Three.js 0.184}: renderizado 3D, shaders GLSL personalizados, EffectComposer.
    \item \textbf{@mediapipe/tasks-vision}: HandLandmarker y FaceLandmarker ejecutados en WASM + WebGL.
    \item \textbf{Web Speech API}: reconocimiento de voz nativo del navegador.
\end{itemize}

\subsection{Arquitectura del Código}

La arquitectura utiliza el patrón de custom hooks React, cada uno encapsulando una modalidad de entrada:

\begin{itemize}
    \item \textbf{\texttt{useHandTracking}}: ejecuta el loop de detección de manos a $\approx$30\,fps y llama a los tres detectores de gestos.
    \item \textbf{\texttt{useFaceTracking}}: gestiona FaceLandmarker, llama a \texttt{expressionDetector} y activa memes o Spider-Sense.
    \item \textbf{\texttt{useSpeechRecognition}}: encapsula Web Speech API; solo escucha cuando el gesto de paz está activo.
    \item \textbf{\texttt{useThreeScene}}: inicializa y conecta el \texttt{SceneManager} con el ciclo de vida React.
\end{itemize}

El estado global se comparte mediante \texttt{AppContext}, que contiene los efectos activos y referencias de alta frecuencia a los landmarks actuales. Todos los efectos Three.js siguen una interfaz uniforme: \texttt{constructor()}, \texttt{init(scene)}, \texttt{update(deltaTime, landmarks)}, \texttt{setActive(bool)}, \texttt{dispose()}.

\subsection{Efectos Implementados}

\subsubsection{Portal Dr.~Strange}

El gesto bimanual activa \texttt{PortalEffect}: un disco Three.js con un shader GLSL de torbellino (\textit{swirl}). El shader anima la distorsión radial de una textura de póster de película en función del tiempo mediante uniforms, creando el efecto característico del portal del Doctor Strage. Un anillo exterior de partículas rodea el disco.

\subsubsection{Tercer Ojo (Snap Aura)}

La aproximación de índice y pulgar activa dos efectos combinados: \texttt{SnapAuraEffect} muestra una esfera brillante verde con un ojo 3D animado posicionado sobre el rostro del usuario (posición calculada convirtiendo landmarks de MediaPipe a coordenadas del mundo Three.js con \texttt{normalizedToWorld()}), y \texttt{ParallaxEffect} genera ojos flotantes que siguen el movimiento de la cabeza con efecto de parallax facial.

\subsubsection{Spider Sense}

La expresión escéptica (debounce de 800\,ms) activa \texttt{SpiderSenseEffect}: líneas de energía generadas desde landmarks distribuidos de la frente del usuario hacia el exterior del canvas, animadas con oscilación sinusoidal. Incluye audio de activación y fade-out suave al desactivarse.

\subsubsection{Efectos de Voz}

Los tres comandos producen efectos diferenciados:
\begin{itemize}
    \item \textbf{Chisme Potente}: aplica \texttt{GlitchPass} del EffectComposer a toda la escena durante 3 segundos, acompañado de audio de corneta pre-decodificado con Web Audio API.
    \item \textbf{Caliente}: overlay CSS con imagen \texttt{hot.png}, animación \textit{bounce-in} de escala y pulso de brillo naranja/rojo, más audio de explosión.
    \item \textbf{Super Elegante}: \texttt{MemeOverlay} con \texttt{ariel.png} en burbuja circular dorada con animación CSS en la esquina inferior derecha.
\end{itemize}

\subsubsection{Memes Homelander}

FaceLandmarker monitoriza cuatro expresiones de forma continua (con debounce individual). Al detectar cada expresión, \texttt{MemeOverlay} muestra la imagen correspondiente en la burbuja dorada. Solo puede estar activo un meme simultáneamente; el más reciente sustituye al anterior.

\subsection{Integración con OBS}

La aplicación está diseñada para su uso con OBS Studio a través de la funcionalidad de Virtual Camera. El streamer abre la aplicación en Chrome (recomendado por su soporte completo de Web Speech API), activa la Virtual Camera del navegador en OBS, y tiene el canvas completo con todos los overlays disponible como fuente de vídeo capturada.

% ── 6. Dificultades ───────────────────────────────────────────────────
\section{Dificultades Encontradas}

\subsection{Detección de Gestos de Mano}

\textbf{Falsos positivos en el portal.} La primera versión usaba un único detector de mano vertical que se activaba con movimientos cotidianos. Se resolvió cambiando a un gesto bimanual que requiere simultáneamente un puño cerrado en la izquierda y todos los dedos extendidos en la derecha, reduciendo drásticamente los falsos positivos al exigir dos condiciones independientes.

\textbf{Handedness invertido.} Con cámara frontal en modo espejo, los labels \textit{Left}/\textit{Right} de MediaPipe están especulares respecto al usuario. Fue necesario intercambiar los labels manualmente en el detector para que el gesto del portal funcionara con la mano correcta.

\textbf{Bug lógico en \texttt{verticalHandDetector}.} La llamada a \texttt{detect()} estaba situada en el bloque \texttt{else} (sin manos presentes), por lo que nunca se ejecutaba cuando había manos detectadas. El portal nunca se activaba. Detectado mediante depuración con \texttt{console.log}.

\textbf{Gesto L poco robusto.} El gesto L original para activar la voz tenía alta tasa de falsos positivos. Se sustituyó por el símbolo de paz (dos dedos en V), con mejor cobertura en el modelo de MediaPipe.

\textbf{Detector de triángulo desde cero.} Requirió múltiples iteraciones con logs de distancias entre landmarks para calibrar los umbrales de activación correctos.

\subsection{Detección de Expresiones Faciales}

\textbf{Umbral de risa demasiado estricto.} El umbral inicial \texttt{jawOpen > 0.5} exigía una apertura bucal exagerada. Reducirlo a \texttt{jawOpen > 0.2} permitió capturar sonrisas naturales con dientes.

\textbf{Spider-Sense con falsos positivos continuos.} La expresión escéptica se detectaba en cualquier parpadeo. Se implementó un debounce de 800\,ms que exige sostener la expresión antes de lanzar el efecto. Este patrón resultó ser necesario para la mayoría de expresiones del sistema.

\textbf{Ojo del Tercer Ojo en posición incorrecta.} \texttt{SnapAuraEffect} leía \texttt{faceDetections[0].keypoints} (formato de la API antigua Face Detection) cuando la API de FaceLandmarker devuelve un array de 478 landmarks con estructura diferente. El ojo se posicionaba en \texttt{(0,0,0)} en lugar de sobre el rostro. La depuración requirió \texttt{console.log} del formato de los datos y ajuste empírico de los índices de landmarks hasta lograr la posición correcta.

\textbf{Rayos del Spider-Sense superpuestos.} Los landmarks de la cabeza están concentrados en la frente, generando rayos en posiciones muy similares que se solapan formando una X en lugar de distribuirse uniformemente. Se resolvió usando un subconjunto distribuido de landmarks con separación angular mínima entre rayos.

\textbf{Grosor de línea en WebGL.} La API de Three.js (\texttt{LineBasicMaterial}) no admite grosor de línea real en WebGL: siempre renderiza a 1\,px independientemente del valor configurado. Los rayos del Spider-Sense no tienen el grosor visual deseado; es una limitación de la plataforma no resoluble sin usar geometría de cilindros o librerías como \texttt{THREE.MeshLine}.

\subsection{Audio y Post-proceso}

\textbf{Póster del portal invisible.} \texttt{renderOrder = -1} situaba el disco del portal detrás del plano de vídeo de la webcam. Cambiado a \texttt{renderOrder = 1} para renderizarlo sobre la escena.

\textbf{Portal no visible en el HUD.} El estado \texttt{verticalHandActive} no estaba registrado en \texttt{ModalitiesPanel}, por lo que el HUD nunca mostraba su estado activo.

\textbf{Audio ausente en Spider-Sense.} El efecto se activaba y desactivaba abruptamente sin ningún audio. Se añadieron sonido de activación y transición de fade-out mediante Web Audio API.

\subsection{Reconocimiento de Voz}

\textbf{Latencia inherente de la Web Speech API.} La API introduce un retraso de $\approx 0.5$--$1\,$s entre la pronunciación y la acción. Es una limitación conocida del navegador, reducible pero no eliminable sin usar reconocimiento local (Whisper.js u otras alternativas), lo que requeriría un backend o descargar modelos de gran tamaño al cliente.

\subsection{Rol del \texttt{.env} en la Depuración}

El archivo \texttt{.env} resultó fundamental para gestionar umbrales configurables (valores de blendshapes, distancias de gestos, duraciones de efectos, flags de debug) sin necesidad de recompilar. Permitió iterar rápidamente sobre los valores óptimos de detección durante el desarrollo, y combinado con el patrón de debounce, fue la herramienta central para distinguir intenciones reales de movimientos accidentales.

% ── 7. Conclusiones ───────────────────────────────────────────────────
\section{Conclusiones}

MoveStream demuestra que es posible construir una interfaz multimodal funcional para streaming completamente en el navegador, sin hardware adicional y con tecnologías de código abierto. Los tres canales de entrada operan de forma combinada y complementaria: el gesto de paz actúa como modificador que habilita el canal de voz, ilustrando el concepto de interacción multimodal cooperativa donde una modalidad puede condicionar y enriquecer a las demás.

Los principales aprendizajes del proyecto son:
\begin{itemize}
    \item \textbf{Los umbrales son críticos}: la detección de gestos y expresiones requiere calibración empírica. Mantener umbrales en variables de entorno configurables es imprescindible para iterar con agilidad.
    \item \textbf{El debounce es imprescindible}: sin él, los efectos se activan continuamente ante cualquier movimiento involuntario, haciendo la experiencia inutilizable en la práctica.
    \item \textbf{Las limitaciones de plataforma importan}: WebGL no admite líneas con grosor real; la Web Speech API tiene latencia inherente. Conocer estas restricciones desde el inicio evita tiempo de depuración innecesario.
    \item \textbf{La multimodalidad aporta valor real}: la combinación de gestos, voz y expresiones faciales genera un vocabulario de interacción mucho más rico y natural que cualquier modalidad por separado, especialmente en el contexto del streaming donde la espontaneidad y la conexión con la audiencia son el valor central.
\end{itemize}

Como trabajo futuro sería valioso reducir la latencia de voz con reconocimiento local (Whisper.js), añadir calibración por usuario de los umbrales gestuales, y explorar la integración directa con la API WebSocket de OBS para control bidireccional entre la aplicación y el software de transmisión.
